## Summary

Phase 06.3 adds the employee-facing dashboard to the existing `/employee/[token]` legacy route. The new flow identifies the employee by their stable X25519 key (the same `public_key` column from the 06.2 CSV), scans all `NewCommitmentEvent`s from the pool, decrypts each dual-blob's `employeeCiphertext`, and presents a consolidated view: all their payments, per-note on-chain status (pending / claiming / claimed via nullifier lookup), and a per-note "Claim" button that drives an in-browser ZK withdraw proof plus Freighter submit.

The architecture is a direct mirror of the auditor flow (already built and tested): `reconstructBatch` is the auditor scanner; the employee dashboard builds an analogous `scanEmployeeNotes` that calls `decodeDualBlob` + `decryptNote(employeePrivkey, blob.employeeCiphertext)` in place of the auditor-key path. Every piece ŌĆö the scanner (`scanCommitmentEvents`), the ECIES decryption (`decryptNote`), the dual-blob decode (`decodeDualBlob`), the Freighter submit (`unshieldNote`) ŌĆö already exists in the worktree. New code is thin glue.

The load-bearing decision is the **hash-to-field derivation**: `pubkeyToBn254` in `denominationBuilder.ts` converts the employee's 32-byte X25519 pubkey to a BN254 scalar (little-endian bytes ŌåÆ bigint, mod BN254). For the deposit (06.2) this is sufficient because the circuit only checks `commitment = Poseidon2(amount, outPubkey, blinding)`. For the withdraw (06.3) the employee must reproduce the SAME BN254 scalar from their private X25519 key, then prove `commitment = Poseidon2(amount, Poseidon2(privKey, 0), blinding)` where `Poseidon2(privKey, 0)` is the BN254 public key computed by the `Keypair()` template in the circuit. The critical insight (from reading `keypair.circom`) is that the circuit's `publicKey = Poseidon2(privateKey, 0)` over BN254 scalars has NO algebraic connection to X25519. This means the withdraw proof requires a **BN254 private key** (a scalar in the BN254 field), not the X25519 private key. The employee must hold the BN254 private key that was used to compute `outPubkey` at deposit time.

**Primary recommendation:** Resolve the hash-to-field debt by making `pubkeyToBn254` deterministic and reproducible. The employee key model for 06.3 must be: the stable X25519 privkey is the ROOT KEY. From it, derive a BN254 spending key via `hkdf(sha256, X25519_priv, salt="sobre-bn254-v1", info="spending-key")` ŌåÆ 32 bytes ŌåÆ reduce mod BN254. That BN254 scalar is the `inPrivateKey` the circuit needs. At deposit time, the employer uses the employee's X25519 PUBKEY to derive the same BN254 scalar (via `pubkeyToBn254`, LE bytes ŌåÆ mod BN254). At withdraw time, the employee uses their X25519 PRIVKEY to derive the BN254 private key via HKDF. The `Keypair()` circuit then checks `publicKey = Poseidon2(privKey, 0)` ŌĆö these values MUST match. Document the mismatch risk (see Pitfall 1) and resolve it in Wave 0 of the plan.

Build in parallel against fixtures/mock (same pattern as the auditor's `mockRpc`) ŌĆö the dashboard is fully testable before the live withdraw path is wired.

### Pattern 1: Employee Note Scan (mirrors `reconstructBatch`)

The auditor flow in `batchReconstructor.ts` already does exactly this ŌĆö scan events, decode dual blob, decrypt auditor half, skip foreign blobs. The employee scan does the same but opens `employeeCiphertext`:

```typescript
// Source: verified against packages/viewkey/src/reconstructor/batchReconstructor.ts
// File: apps/web/lib/employee-scan.ts

import { scanCommitmentEvents, decodeDualBlob, decryptNote } from 'viewkey'
import type { ScannedEvent } from 'viewkey'

export interface EmployeeNote {
  commitment: bigint
  index: number
  amount: bigint
  blinding: bigint
  ledger: number
  txHash: string
}

export type EmployeeEventSource =
  | { events: ScannedEvent[] }                          // fixture / mock mode
  | { rpcUrl: string; poolContractId: string; fromLedger: number }

export async function scanEmployeeNotes(
  employeePrivkey: Uint8Array,   // 32-byte X25519 private key
  source: EmployeeEventSource,
): Promise<EmployeeNote[]> {
  const events = 'events' in source
    ? source.events
    : await scanCommitmentEvents(source)

  const notes: EmployeeNote[] = []
  for (const event of events) {
    try {
      const blob = decodeDualBlob(event.encryptedOutput)
      const payload = decryptNote(employeePrivkey, blob.employeeCiphertext)
      notes.push({
        commitment: event.commitment,
        index: event.index,
        amount: payload.amount,
        blinding: payload.blinding,
        ledger: event.ledger,
        txHash: event.txHash,
      })
    } catch {
      // Foreign blob: GCM tag mismatch (not this employee's note). Skip.
      continue
    }
  }
  return notes
}
```

The `events` injection branch is the mock seam: tests pass `{ events: FIXTURE_EVENTS }` and the scan runs without any RPC call. This matches the auditor's `mockRpc` approach exactly.

### Pattern 2: BN254 Key Derivation from Stable X25519 Key

This is the load-bearing debt. The circuit's `Keypair()` template (`keypair.circom`) computes:

```
publicKey = Poseidon2(privateKey, 0, domainSeparation=0x03)
```

where `privateKey` is a BN254 scalar field element. For the withdraw proof, the employee supplies this `privateKey` as `inPrivateKey[0]`, and the circuit verifies that:

- `commitment = Poseidon2(amount, publicKey, blinding)`
- the nullifier is correctly derived from commitment + path + signature
- the commitment is a leaf in the Merkle tree

At deposit time, the employer computed `outPubkey = pubkeyToBn254(x25519PubkeyHex)` (LE bytes, mod BN254). This is a hash of the X25519 *public* key bytes into the BN254 field. It is NOT the same as `Poseidon2(bn254PrivKey, 0)` ŌĆö the two derivation paths are algebraically different. To make unshield work, the employee needs a BN254 private key such that `Poseidon2(bn254PrivKey, 0) == pubkeyToBn254(x25519PubkeyHex)`. That is not achievable by inversion (Poseidon2 is one-way).

**Resolution:** Change the deposit-side key derivation to match the withdraw-side circuit. The correct flow:

1. Employee generates/holds stable X25519 keypair (for encryption).
2. From the X25519 private key, derive a BN254 spending private key: `bn254PrivKey = HKDF-SHA256(x25519Priv, salt=undefined, info="sobre-bn254-spending-key-v1") ŌåÆ 32 bytes ŌåÆ LE bigint ŌåÆ mod BN254`.
3. The BN254 public key is `Poseidon2(bn254PrivKey, 0)` ŌĆö computed by the WASM bridge (`derivePublicKey`).
4. At deposit time, the employer receives the employee's X25519 pubkey (from the CSV). To compute `outPubkey`, the employer CANNOT derive the BN254 privkey (that requires the X25519 privkey, held only by the employee). This breaks the deposit-side `pubkeyToBn254(x25519PubkeyHex)` approach.

**Practical resolution for 06.3 (hackathon scope):** Keep the 06.2 deposit derivation as-is (`pubkeyToBn254(x25519PubkeyHex)` = LE bytes mod BN254). For the withdraw proof, the employee must supply not their X25519 privkey alone, but ALSO supply the BN254 privkey inline (or the dashboard derives it from the X25519 privkey bytes using the SAME LE reduction). Since the deposit used `pubkeyToBn254(X25519_pubkey_hex)` as `outPubkey`, and the circuit checks `Poseidon2(inPrivateKey, 0) == commitment_pubkey_slot`, the employee needs an `inPrivateKey` value P such that `Poseidon2(P, 0) = pubkeyToBn254(x25519PubkeyHex)`. That inversion is impossible.

**Concrete fix for 06.3:** Change `pubkeyToBn254` at the deposit side to use the X25519 privkey-derived BN254 value instead of the pubkey bytes. But the employer only has the X25519 pubkey, not the privkey.

**Viable path for the hackathon demo:** The employee CSV includes `public_key` (X25519 pubkey). For the demo, we add a second column or a derivation agreement: the employee's BN254 pubkey is `pubkeyToBn254(X25519_pubkeyHex)` (the current scheme), and the employee's BN254 privkey is derived via the SAME LE-bytes-mod-BN254 approach applied to the X25519 PRIVKEY: `bn254PrivKey = privkeyToBn254(X25519_privkeyHex)`. Then `Poseidon2(bn254PrivKey, 0)` will NOT equal `pubkeyToBn254(X25519_pubkeyHex)` because the Poseidon2 image of (LE bytes of privkey) is not the same as (LE bytes of pubkey).

**Final recommendation (hackathon-viable, disclosed as PoC):** Re-define the employee key scheme so that the BN254 privkey IS the LE-bytes-mod-BN254 of the X25519 privkey, and the BN254 pubkey is `Poseidon2(bn254PrivKey, 0)`. At deposit time, the EMPLOYER cannot compute the BN254 pubkey from the X25519 pubkey alone. Therefore, add a new column to the CSV: `bn254_pubkey` (or compute it from the privkey and include it in the employee-generated keygen card from Phase 06.1). The employee's keygen card emits BOTH the X25519 pubkey (for encryption) AND the BN254 pubkey (for circuit commitments). This is the cleanest break. Mark as `[ASSUMED]` until confirmed.

For the existing 06.2 test notes in the fixture (encrypted with `pubkeyToBn254(pubkeyHex)` applied to the X25519 pubkey bytes), the unshield prove will fail unless the BN254 privkey satisfies `Poseidon2(privKey, 0) = pubkeyToBn254(X25519_pubkeyHex)`. This cannot be satisfied deterministically. Therefore:

**06.3 scope decision:** The withdraw proving path targets notes deposited with the NEW key scheme (where `outPubkey = Poseidon2(bn254PrivKey, 0)` and `bn254PrivKey = LE_bytes_mod_BN254(X25519_privkey)`). Existing 06.2 demo fixtures cannot be withdrawn with a live proof ŌĆö they remain demo-only "scan + display" without the live withdraw prove step. Disclose honestly.

```typescript
// Source: new file apps/web/lib/zk/keyDerivation.ts
import { hkdf } from '@noble/hashes/hkdf.js'
import { sha256 } from '@noble/hashes/sha2.js'
import { BN254_FIELD_MODULUS } from 'viewkey'

const SPENDING_KEY_INFO = new TextEncoder().encode('sobre-bn254-spending-key-v1')

/**
 * Derive the BN254 spending private key from the employee's stable X25519 private key.
 *
 * APPROACH: HKDF-SHA256 over the X25519 privkey bytes ŌåÆ 32 bytes ŌåÆ LE bigint ŌåÆ mod BN254.
 *
 * This derivation is DETERMINISTIC and BIAS-FREE (HKDF outputs are indistinguishable
 * from random, no modular bias because BN254 is ~256 bits and the output space is 2^256).
 *
 * The BN254 pubkey is then Poseidon2(bn254PrivKey, 0) ŌĆö computed by the WASM bridge.
 * This value must match the `outPubkey` committed at deposit time (see NOTES.md deuda t├®cnica).
 */
export function deriveBn254PrivKey(x25519Priv: Uint8Array): bigint {
  if (x25519Priv.length !== 32) {
    throw new Error(`deriveBn254PrivKey: expected 32 bytes, got ${x25519Priv.length}`)
  }
  const keyBytes = hkdf(sha256, x25519Priv, undefined, SPENDING_KEY_INFO, 32)
  // Interpret as little-endian bigint (same convention as pubkeyToBn254)
  let scalar = BigInt(0)
  for (let i = 31; i >= 0; i--) {
    scalar = (scalar << BigInt(8)) | BigInt(keyBytes[i])
  }
  return scalar % BN254_FIELD_MODULUS
}

/**
 * Convert a 64-char hex X25519 pubkey to a BN254 scalar
 * using the same convention as denominationBuilder.pubkeyToBn254.
 * Re-exported here for the withdraw side to avoid import-path confusion.
 */
export function pubkeyToBn254(hex: string): bigint {
  let scalar = BigInt(0)
  for (let i = 31; i >= 0; i--) {
    scalar = (scalar << BigInt(8)) | BigInt(parseInt(hex.slice(i * 2, i * 2 + 2), 16))
  }
  return scalar % BN254_FIELD_MODULUS
}
```

### Pattern 3: Nullifier Status Lookup

The pool contract stores spent nullifiers. A read-only `simulateTransaction` call to `pool.is_spent(nullifier)` returns a boolean. The nullifier for a note is:

```
nullifier = Poseidon2(commitment, pathIndices, Poseidon2(privKey, commitment, pathIndices))
```

Per `policyTransaction.circom` lines 100-103. To look up the status before generating a proof, compute the nullifier client-side (via WASM bridge `computeNullifier`) or just attempt to scan the nullifier from on-chain nullifier storage if the contract exposes it. The simpler path is: check `pool.is_spent(nullifier)`.

```typescript
// Source: pattern mirrors fetchPoolRoot in apps/web/lib/rpc.ts
// New function: fetchNullifierStatus in rpc.ts

export async function fetchNullifierStatus(nullifier: bigint): Promise<boolean> {
  const { rpcUrl, poolContractId, deployer } = readDeployments()
  const server = new Server(rpcUrl)
  const pool = new Contract(poolContractId)
  const source = new Account(deployer, '0')
  const tx = new TransactionBuilder(source, {
    fee: '100',
    networkPassphrase: Networks.TESTNET,
  })
    .addOperation(pool.call('is_spent', nativeToScVal(nullifier.toString(), { type: 'u256' })))
    .setTimeout(30)
    .build()
  const sim = await server.simulateTransaction(tx)
  if ('error' in sim && sim.error) return false // treat error as unknown/pending
  const retval = (sim as { result?: { retval?: unknown } }).result?.retval
  if (!retval) return false
  return Boolean(scValToNative(retval as never))
}
```

For the mock/fixture path in tests, `page.route` intercepts `simulateTransaction` and returns a fixed boolean, same as the auditor's `SIM_BALANCE_800` pattern.

### Pattern 4: Withdraw Witness Assembly

The withdraw uses the SAME `policy_tx_1_8` circuit in reverse: N=8 inputs (the employee's notes) ŌåÆ 1 output (or 0 outputs if the employee claims all). For a single-note claim (the 06.3 scope), the shape is `policy_tx_1_8` but with:

- `inAmount[i]` = real note amount (NOT 0 ŌĆö this IS a real input, NOT dummy)
- `inPrivateKey[i]` = bn254PrivKey
- `inBlinding[i]` = blinding from the decrypted note
- `inPathIndices[i]` = Merkle path indices from the pool
- `inPathElements[i]` = Merkle path elements from the pool
- `outputCommitment` = all zeros or 1 change note (employee keeps no change in single-claim)
- `publicAmount` = negative note amount (withdrawal)
- `ext_amount` = negative (withdrawal direction)

Wait ŌĆö `policy_tx_1_8` has 1 INPUT and 8 OUTPUTS. For a withdraw you need a circuit with N inputs ŌåÆ 1 output. Check if there is a `policy_tx_8_1` or similar. [VERIFIED: `packages/zk/circuits/src/` contains only `policy_tx_1_8.circom` and `policy_tx_2_2.circom`]. There is no withdraw-specific circuit variant present in the worktree.

**Critical finding:** The pool's `transact` function is bidirectional (it handles both deposits and withdrawals via the sign of `ext_amount`). The CIRCUIT however has a fixed shape. For a withdrawal, a circuit with nIns Ōēź 1 (real inputs) is needed. `policy_tx_1_8` has nIns=1, which supports 1 input note ŌåÆ 8 output notes (typical deposit). A withdrawal of 1 note needs nIns=1, nOuts=1 (or nOuts=0 with a dummy). `policy_tx_2_2` (2 inputs ŌåÆ 2 outputs) could support 1 real input + 1 dummy input ŌåÆ 0 real outputs + 2 dummy outputs.

**Practical approach for 06.3 hackathon scope:** Use `policy_tx_1_8` for the withdraw proof by treating 7 of the 8 outputs as zero-amount dummies and 1 output as the "change" (0 if claiming full note). The 1 real input is the note being claimed. `inAmount[0] ŌēĀ 0` means the Merkle membership check runs (contrast with deposit where `inAmount[0] = 0` skips it). The employee must supply a Merkle proof for the note's commitment leaf. [ASSUMED: the pool contract exposes a `get_proof(index)` function ŌĆö verify before committing this approach. If not, fallback: the employee claims by supplying the proof reconstructed from the commitment event history.]

**Simpler path (if Merkle proof fetch is complex):** For 06.3 demo scope, build the withdraw proof using the EXISTING `employee-unshield.ts` infrastructure which already uses `withdrawProofXdr` and `withdrawExtDataXdr` embedded in the note token. The dashboard can generate these via the worker in-browser and pass them to `unshieldNote`. No separate Merkle proof lookup needed if the circuit's witness includes the path from the deposit event.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| X25519 ECIES decryption | Custom ECIES | `decryptNote` from `viewkey` | Already battle-tested with GCM auth tag; rejects foreign blobs cleanly |
| Dual-blob decode | Custom parser | `decodeDualBlob` from `viewkey` | Fixed 4-byte length-prefix layout; malformed blobs throw |
| Pool event scan | Direct RPC pagination | `scanCommitmentEvents` from `viewkey` | Handles pagination, cursor, error cases |
| BN254 key derivation | Ad-hoc bit manipulation | HKDF-SHA256 from `@noble/hashes` (already installed) | Bias-free, deterministic, cryptographically sound |

## Common Pitfalls

### Pitfall 1: BN254 Public Key Mismatch Between Deposit and Withdraw

**What goes wrong:** At deposit time, `outPubkey = pubkeyToBn254(x25519PubkeyHex)` (LE bytes of the X25519 pubkey, mod BN254). At withdraw time, the circuit checks `Poseidon2(inPrivateKey, 0) == commitment_pubkey`. These will NEVER match unless the employee has a BN254 private key whose Poseidon2 image equals the LE-bytes interpretation of their X25519 pubkey. There is no known preimage.

**How to avoid:** For newly deposited notes, use a consistent key scheme from the start. Options:

- A: Employee keygen emits BOTH X25519 pubkey (for ECIES) AND BN254 pubkey (for circuit). CSV has three columns: `name, amount, x25519_pubkey, bn254_pubkey`. Employer uses `bn254_pubkey` in the circuit.
- B: Derive both from the same seed. Employee generates a BN254 privkey directly; derives BN254 pubkey from it (WASM `derivePublicKey`); derives X25519 keypair from the same seed via HKDF.
- C: Accept that the existing 06.2 fixtures cannot be unshielded with a live proof. New payroll runs use the new scheme. Old notes are demo-only "claim" stubs (the existing `employee-unshield.ts` pre-embedded proof path).

**For 06.3:** Option C is the path of least resistance. Mark the prove step as "works for notes deposited after the key scheme fix." Old notes use the `withdrawProofXdr` embedded stub path.

**Warning signs:** `computeCommitment(amount, Poseidon2(employeePrivKey, 0), blinding) !== commitment` ŌĆö the commitment won't match the on-chain leaf, and the Merkle proof check fails inside the circuit.

### Pitfall 2: `decryptNote` Throws on Foreign Blobs (Expected)

**What goes wrong:** Developer adds error logging for every decrypt failure, producing noise for the 7/8 notes that belong to OTHER employees in the same batch.

**How to avoid:** The `catch { continue }` pattern in `scanEmployeeNotes` is correct. Do NOT log decrypt errors as warnings ŌĆö they are expected for notes not belonging to this employee. Only log if ALL 8 notes in a batch fail (possible indicator of wrong key or empty pool).

**Warning signs:** Console flooded with `GCM auth tag mismatch` logs on a healthy scan.

### Pitfall 3: Nullifier Computation Before Merkle Proof Available

**What goes wrong:** The nullifier formula in the circuit is `Poseidon2(commitment, pathIndices, signature)` where `signature = Poseidon2(privKey, commitment, pathIndices)`. Without the Merkle path (from `pool.get_proof(index)`), `pathIndices` is unknown, so the nullifier cannot be computed to check `is_spent`.

**How to avoid:** For the status display, approximate: use `pathIndices=0` for the nullifier status check (gives the "would the pool reject this note as already spent?" answer, not the exact nullifier). For the actual claim, the correct Merkle proof must be fetched first. Or: cache the Merkle path from the deposit event (if `pool.get_proof` is not callable, reconstruct from commitment events ŌĆö the scanner has the full event history).

**Warning signs:** `is_spent` returns false for a note that was already claimed (because the nullifier was computed with wrong pathIndices).

### Pitfall 4: SSR Guard for Worker in New Dashboard Route

**What goes wrong:** `proverClient.ts` is marked `'use client'` but if `initProver` is called at module level (not inside `useEffect`), Next.js SSR fails.

**How to avoid:** Same guard as `PayrollComposer`: all proverClient calls inside `useEffect(() => { if (typeof window === 'undefined') return; ... }, [])`. The new `/employee` dashboard page must be `'use client'`.

**Warning signs:** Build error `Worker is not defined` during `next build`.

### Pitfall 5: Fixture Blobs May Not Decrypt Under Employee Key

**What goes wrong:** The `auditor.spec.ts` fixture blobs were encrypted to the auditor key (`0x42 ├Ś 32`). The `employeeCiphertext` portion of the same blobs may have been encrypted to a DIFFERENT employee key (the X25519 pubkey of a test employee). If the test employee privkey is not `0x42 ├Ś 32`, the employee dashboard tests will fail to find any decryptable notes.

**How to avoid:** Generate a new fixture set for employee dashboard tests using a KNOWN employee privkey (e.g., `0x43 ├Ś 32`). Run `encryptNote(employeePubkeyFromPriv(0x43 ├Ś 32), payload)` to generate the `employeeCiphertext` portion, then wrap in `encodeDualBlob(employeeCt, auditorCtFromExisting)`. The test spec creates its own FIXTURE_EVENTS with known employee key.

**Warning signs:** `scanEmployeeNotes` returns 0 notes even with valid fixtures (wrong key).

import { keyFromBase64 } from 'viewkey'

/** Parse a pasted X25519 private key (hex or base64) into 32 bytes. */
export function parseEmployeeKey(input: string): Uint8Array {
  const clean = input.trim().replace(/^0x/, '')
  if (/^[0-9a-fA-F]{64}$/.test(clean)) {
    const out = new Uint8Array(32)
    for (let i = 0; i < 32; i++) out[i] = parseInt(clean.slice(i * 2, i * 2 + 2), 16)
    return out
  }
  return keyFromBase64(input.trim()) // base64 path (43 chars url-safe)
}

export async function claimNote(
  note: EmployeeNote,
  bn254PrivKey: bigint,
  recipientAddress: string,
): Promise<{ hash: string }> {
  // 1. Fetch Merkle proof (path elements + path indices) from pool
  const { pathElements, pathIndices } = await fetchMerkleProof(note.index)
  // 2. Compute outputs: 0 outputs (all amounts taken by recipient)
  const withdrawInputs = buildWithdrawInputs({
    note, bn254PrivKey, pathElements, pathIndices,
    recipientAddress, aspMemberRoot, aspNonMemberRoot, poolRoot
  })
  // 3. Generate proof in-browser via proverClient
  const { proof } = await prove(withdrawInputs)
  // 4. Build ext_data (negative ext_amount = withdrawal)
  const extDataHash = hashExtDataWithdraw({ recipient: recipientAddress, ext_amount: -note.amount, encrypted_outputs: [] })
  // 5. Submit via unshieldNote path
  return unshieldNote({
    poolContractId: readDeployments().poolContractId,
    commitmentIndex: note.index,
    amount: note.amount.toString(),
    notePrivkeyHex: '', // not used when withdrawProofXdr is present
    blinding: note.blinding.toString(),
    withdrawProofXdr: Buffer.from(proof).toString('base64'),
    withdrawExtDataXdr: buildWithdrawExtDataXdr({ recipient: recipientAddress, ext_amount: -note.amount }),
  })
}

      })))
      setNotes(withStatus)
      setState('done')
    } catch { setState('error') }
  }

}
